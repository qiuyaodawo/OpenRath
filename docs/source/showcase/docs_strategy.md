# 文档展示策略

目标是让读者按意图进入不同深度，逐步接触源码细节。

## 首页应该承担什么

首页只回答四个问题：

1. OpenRath 是什么？
2. 它和普通 chat wrapper 的工程差异是什么？
3. 最短代码怎么跑？
4. 下一步去哪里？

首页不应该解释所有类和所有字段。否则读者还没建立兴趣就被细节淹没。

## 用户指南应该承担什么

用户指南按运行路径写：

```text
Concepts → Session → Backend → Tools → Workflow → LLM
```

这比按源码目录更好，因为用户先关心“怎么跑起来”和“状态怎么流动”，不是先关心模块文件名。

## API 参考应该承担什么

API 参考应该短而准：

- 每个包导出什么；
- 每个子模块负责什么；
- 关键签名是什么；
- 哪些旧接口不存在。

不要在 API 参考里写长篇教程。教程放用户指南，故事放 showcase。

## 展示材料应该承担什么

展示材料要面向不同人群：

| 人群 | 关心点 | 推荐文案 |
| --- | --- | --- |
| 工程师 | API 是否干净、是否可运行 | Quickstart + PyTorch 类比 |
| 投资人/合作方 | 为什么重要、能扩展到哪里 | 一页纸介绍 + 架构故事 |
| 开源用户 | 怎么开始、怎么贡献 | README + examples |
| 内部团队 | 下一步怎么写文档 | 本策略页 |

## 当前最该优先完善的三件事

1. README：把定位和最小 demo 写清楚。
2. Tools 文档：严格解释 `FlowToolCall`，避免旧 `ToolTable` 误导。
3. Examples：每个示例都说明依赖、运行命令、可能踩坑。
